# PostgreSQL Generated Columns 개념 정리

작성일: 2026-04-22

## 개념

PostgreSQL 12+ 에서 지원하는 **자동 계산 컬럼**. 컬럼 값을 다른 컬럼의 식으로 선언하면 DB가 INSERT/UPDATE 시 자동으로 계산해서 저장한다.

```sql
ALTER TABLE match
ADD COLUMN tier_bucket INTEGER
GENERATED ALWAYS AS (
    CASE
        WHEN average_tier >= 10000 THEN 10000
        ELSE (average_tier / 1000) * 1000
    END
) STORED;
```

CHALLENGER(기준 10000)는 LP 상한이 없어 `average_tier` 가 11022, 15000 처럼 나올 수 있으므로 10000에서 캡을 씌워 모든 챌린저 참가자를 하나의 버킷으로 묶는다. 캡 없이 단순 절삭하면 `11022 → 11000`, `15000 → 15000` 식으로 챌린저가 여러 버킷으로 쪼개져 집계가 깨진다.

이후:
```sql
INSERT INTO match (average_tier) VALUES (10549);
-- PG가 자동으로 tier_bucket = 10000 계산해서 저장
INSERT INTO match (average_tier) VALUES (11022);
-- CHALLENGER 캡으로 tier_bucket = 10000 (11000 아님)
```

앱 코드는 원본 컬럼만 넣으면 되고, 파생값은 DB가 보장.

## STORED vs VIRTUAL

SQL 표준은 두 종류를 정의함:

| 종류 | 동작 | PG 지원 |
|---|---|---|
| **STORED** | 저장 시 계산 → 디스크에 실제 저장. 인덱스 가능 | O |
| **VIRTUAL** | 조회 시 매번 계산. 저장 안 됨 | X (MySQL은 지원) |

**PG는 STORED만 지원**. VIRTUAL 이 필요하면 `VIEW` 로 대체.

## `GENERATED ALWAYS` vs `GENERATED BY DEFAULT`

- **`GENERATED ALWAYS`**: 수동 INSERT/UPDATE 값 지정 **불가**. 무조건 식 기반. (대부분의 경우 이걸 씀)
- **`GENERATED BY DEFAULT`**: 수동 값 지정 시 override 가능. identity 컬럼(IDENTITY) 에서 주로 쓰이는 옵션이고, generated 컬럼에는 사실상 `ALWAYS` 만 사용.

※ `GENERATED ... AS IDENTITY` (자동 증가 컬럼) 와 `GENERATED ... AS (식) STORED` (파생 컬럼) 는 문법만 비슷하지 다른 기능.

## 장점

1. **앱 코드 변경 불필요** — 스키마만 바꾸면 끝. 모든 INSERT 경로가 자동 수혜
2. **일관성 보장** — 수동 UPDATE 자체 불가능해서 원본과 어긋날 수 없음
3. **인덱스 가능** — 일반 컬럼과 동일하게 `CREATE INDEX ON t (tier_bucket)` 사용 가능
4. **선언적** — 테이블 스키마만 봐도 파생 규칙이 명확. 트리거처럼 "숨은 로직" 없음
5. **쿼리 성능** — 매번 계산하는 대신 저장된 값 읽으므로 집계/조인 시 이점

## 제약

### 식은 IMMUTABLE 해야 함
- 허용: 산술, 문자열 함수, `CASE`, 내장 immutable 함수
- 금지: `now()`, `random()`, `current_user`, 다른 테이블 조회, 사용자 정의 volatile 함수

이유: 저장 시점에 값이 고정되어야 하는데, 식 결과가 시간/환경에 따라 달라지면 데이터 일관성이 깨짐.

### 다른 generated 컬럼 참조 불가
```sql
-- 이건 안 됨:
ADD COLUMN b INTEGER GENERATED ALWAYS AS (a + 1) STORED,
ADD COLUMN c INTEGER GENERATED ALWAYS AS (b + 1) STORED;  -- b는 generated
```

### 기존 테이블에 ADD COLUMN 시 rewrite 발생
- `AccessExclusiveLock` 걸림 → 테이블 전체 잠금
- 행이 많으면 수 분~수 시간 소요
- **대형 테이블 무중단 추가 전략**:
  1. 일반 NULL 컬럼 추가
  2. 트리거로 신규/수정 행은 자동 채움
  3. 배치로 기존 행 백필
  4. NOT NULL 제약 추가
  5. (선택) 트리거 제거하고 generated 로 전환은 또 rewrite 필요하므로, 대형 테이블은 그냥 트리거로 유지하는 경우도 있음

### 원본 컬럼 UPDATE → 행 rewrite
Generated 컬럼 때문이 아니라 PG MVCC 의 일반 동작. `average_tier` 를 UPDATE 하면 새 행 버전이 만들어지고 `tier_bucket` 도 함께 재계산됨.

## 대안 비교

| 방식 | 자동 일관성 | 인덱스 가능 | 복잡도 | 비고 |
|---|---|---|---|---|
| `GENERATED STORED` | O | O | 낮음 | 선언적, 단순 파생에 최선 |
| 트리거 (BEFORE INSERT/UPDATE) | O | O | 중간 | 유연하지만 보일러플레이트. 누락 위험 |
| 뷰에 계산 컬럼 | O | X | 낮음 | 저장 안 됨. 복잡한 집계에 적합 |
| 앱 레이어 계산 후 INSERT | X | O | 중간 | 누락/불일치 위험. 쿼리마다 중복 |

**단순 파생값(산술, 문자열 조합, CASE)** 이면 Generated 가 거의 항상 정답.
**복잡한 로직이나 외부 참조 필요** 하면 트리거 또는 뷰로.

## 예시

### 산술 파생 + 상한 캡
```sql
ADD COLUMN tier_bucket INTEGER
GENERATED ALWAYS AS (
    CASE
        WHEN average_tier >= 10000 THEN 10000
        ELSE (average_tier / 1000) * 1000
    END
) STORED;
```
LoL 티어 버킷 — CHALLENGER 의 LP 상한 없음을 처리하기 위해 10000 캡.

### 정규화 / 문자열 조합
```sql
ADD COLUMN full_name TEXT
GENERATED ALWAYS AS (first_name || ' ' || last_name) STORED;
```

### CASE 분기
```sql
ADD COLUMN tier_group VARCHAR(10)
GENERATED ALWAYS AS (
    CASE
        WHEN average_tier >= 8000 THEN 'HIGH'
        WHEN average_tier >= 4000 THEN 'MID'
        ELSE 'LOW'
    END
) STORED;
```

### 부분 인덱스 대상으로 활용
```sql
CREATE INDEX idx_match_high_tier ON match (match_id)
WHERE tier_bucket >= 8000;
```

## 요약

- **Generated STORED = 트리거의 선언적 / 안전한 버전**. 단순 파생값엔 이게 기본값
- 식이 IMMUTABLE 이어야 하고, 기존 대형 테이블 추가는 rewrite 비용 주의
- PG는 VIRTUAL 미지원이라 "매번 계산" 목적은 뷰로 대체
- 인덱스 가능 + 일관성 자동 보장이 핵심 장점
