# 파티셔닝 (Partitioning) 개념 정리

작성일: 2026-04-22

## 개념

**파티션 컬럼 값 기준으로 테이블을 별도 저장 단위로 분할** 하는 물리적 조직 방식.
논리 테이블은 하나지만 내부적으로 여러 물리 파티션으로 나뉘어 있고, 쿼리 엔진이 필터 조건을 보고 **관련 파티션만 읽는다** (나머지는 디스크 접근조차 안 함).

파일시스템에서 폴더로 나눈 것과 유사:
```
patch_date=2026-04-01/
patch_date=2026-04-15/
patch_date=2026-04-22/
```

## 파티션 프루닝 (Partition Pruning)

쿼리의 `WHERE` 절에 파티션 컬럼 필터가 있으면 엔진이 **해당 파티션 외에는 스캔하지 않음**.

```sql
SELECT * FROM match
WHERE patch_date = '2026-04-22'
```
→ 100개 패치 중 1개 파티션만 읽음. 나머지 99% 스킵.

## 엔진별 구현 차이

### BigQuery
- 단일 파티션 컬럼만 허용 (보통 DATE/TIMESTAMP 또는 정수 range)
- 파티션 단위 = 보통 하루 / 정수 범위
- `PARTITION BY DATE(created_at)` 형태로 선언
- `_PARTITIONDATE` 같은 가상 컬럼 제공
- 과금: 스캔한 바이트 기준이라 **프루닝 = 비용 절감** 이 직접적

### PostgreSQL
- 선언적 파티셔닝 (10+) — RANGE / LIST / HASH 지원
- 부모 테이블에 자식 파티션 `ATTACH`
- 실행 계획에서 `Partition Pruning` 노드 확인 가능
- 주 목적: 대형 테이블 관리 (오래된 파티션 DROP 용이, VACUUM 분산)

### 분석 DW (Snowflake, Redshift 등)
- 별도의 명시적 파티션 선언 없이 **자동 마이크로파티션** 으로 관리 (Snowflake)
- 또는 SORT KEY 기반 (Redshift — 클러스터링과 경계 모호)

## 왜 유용한가

1. **스캔 범위 축소** → 응답 속도 향상
2. **BQ에서는 과금 직접 절감** (TB 단위 과금이라 수십~수백배 차이)
3. **데이터 수명 주기 관리** — 오래된 파티션 DROP/EXPIRE 용이
4. **병렬 처리** — 파티션 단위로 독립 읽기

## 설계 기준

좋은 파티션 컬럼의 조건:
- **거의 모든 쿼리가 이 컬럼으로 필터** 되어야 함 (그렇지 않으면 프루닝 효과 0)
- **카디널리티 적당해야 함** — 너무 낮으면 분할 의미 없고 (2~3개 파티션), 너무 높으면 파티션 폭발 (BQ는 파티션 수 상한 있음, 4000)
- **시간 기반이 가장 흔함** — 게임 분석이면 `patch_date`, 이벤트 로그면 `event_date`

**안티 패턴**:
- 파티션 컬럼 없이 필터 (전체 스캔)
- 파티션 컬럼을 함수로 감싸기 — `WHERE DATE(created_at) = ...` 대신 `WHERE created_at >= '...' AND created_at < '...'` 써야 프루닝 작동 (엔진별 다름)
- 너무 세밀한 파티션 (1시간 단위 등) — 메타데이터 오버헤드 + 파티션 수 상한 초과

## 주의사항

- **파티션 컬럼을 WHERE에 안 쓰면 프루닝 0** → 풀스캔됨. 팀 쿼리 리뷰 시 1순위 체크 항목
- **동적 파티션 필터** (서브쿼리/조인 결과로 필터링) 은 엔진에 따라 프루닝 못 할 수 있음. 상수 또는 정적 범위로 필터링하는 게 안전
- **파티션 컬럼 변경은 비용 큼** — 테이블 재생성 필요한 경우 많음

## 요약

- **파티셔닝 = 테이블을 조각내서, 조건 맞는 조각만 읽게 하는 구조**
- 엔진 무관하게 핵심은 동일: *어떤 컬럼이 거의 모든 쿼리의 필터 조건으로 쓰이는가*
- BQ에서는 특히 비용 관점에서 파티셔닝 전략이 가장 중요한 설계 결정
