# 클러스터링 (Clustering) 개념 정리

작성일: 2026-04-22

## 개념

**테이블 데이터를 특정 컬럼 기준으로 정렬해서 물리적으로 저장** 하고, 각 저장 블록마다 `min/max` 메타데이터를 함께 유지하는 방식.

쿼리 엔진은 필터 조건을 보고 **블록 메타데이터로 범위 밖 블록을 통째로 스킵** 한다. 인덱스가 아니라 **블록 레벨 스킵 메커니즘**.

```
CLUSTER BY tier_bucket, champion_id 인 경우:

파티션 내부의 블록들:
  block 1: tier_bucket 1000~3000, champion_id 1~50
  block 2: tier_bucket 4000~6000, champion_id 1~50
  block 3: tier_bucket 8000~10000, champion_id 1~200   ← 마스터+
```

쿼리 `WHERE tier_bucket >= 8000` → block 3만 읽고 block 1/2 스킵.

## 동작 원리

1. INSERT 시 (또는 백그라운드 재조직 시) 클러스터 키로 정렬해서 블록에 저장
2. 블록 메타데이터 = 블록 내 각 컬럼의 `min/max`
3. 쿼리 시 필터 조건이 블록의 `[min, max]` 범위와 겹치지 않으면 블록 전체 스킵
4. 살아남은 블록은 **안에서 선형 스캔**

핵심: **O(log n) 탐색이 아니라 "관련 없는 블록을 빠르게 버리는" 메커니즘**.

## 엔진별 구현

### BigQuery
- 최대 4개 컬럼까지 클러스터링 가능
- 백그라운드로 비동기 재조직 (최근 INSERT는 덜 정렬됨)
- `CLUSTER BY col1, col2, ...` 선언
- 파티셔닝과 조합해서 사용 (파티션 내부에서 클러스터링)

### PostgreSQL — BRIN (Block Range INdex)
- 철학상 가장 가까운 동일 개념
- `CREATE INDEX ... USING BRIN (col)`
- 블록 범위별 min/max 저장
- 매우 작은 크기의 인덱스 (테이블의 0.x%)
- 데이터가 물리 순서로 적재되어 있을 때 효과적 (예: 시간순 append-only 로그)

### Redshift / Snowflake / Oracle
- **Zone Maps** 라고 부름 — BQ 클러스터링과 완전히 같은 개념

### SQL Server / MySQL
- **Clustered Index** 가 있지만 B-tree 탐색을 포함한 구조라 의미가 다름 (아래 비교 참조)

## 클러스터 컬럼 순서가 중요한 이유

`CLUSTER BY tier_bucket, champion_id` 인 경우:
- **첫 컬럼 기준으로 1차 정렬** → `tier_bucket` 단독 필터는 매우 효과적
- 같은 `tier_bucket` 내에서 `champion_id` 로 2차 정렬 → `tier_bucket + champion_id` 함께 필터링하면 최고 효과
- **`champion_id` 단독 필터** 는 효과 급감 (블록 전체에 champion_id 가 퍼져있어 스킵 못 함)

**설계 원칙**: 가장 선택도 높고 자주 쓰는 필터 컬럼을 맨 앞에.

## 효과 조건

클러스터링이 효과적이려면:
- **고선택도 필터** (전체의 1% 이하로 좁혀지는 조건)
- **쿼리에 클러스터 컬럼 필터가 실제로 있음**
- **데이터가 클러스터 순서대로 어느 정도 정렬되어 있음**

효과 없는 경우:
- 전체 스캔 쿼리 (어차피 다 읽어야 함)
- 클러스터 컬럼 필터 없는 쿼리
- 클러스터 컬럼 순서 잘못 잡음 (자주 쓰는 컬럼이 뒷순위)
- 데이터가 너무 무작위로 분포 (블록 min/max가 넓어져 스킵 불가)

## 커버링 인덱스 vs 클러스터링 — 혼동 주의

| 항목 | 커버링 인덱스 (RDBMS) | 클러스터링 (BQ/BRIN) |
|---|---|---|
| 자료구조 | B-tree 별도 저장 | 테이블 자체를 정렬 |
| 조회 | `O(log n)` 탐색 | 블록 min/max 로 스킵 후 **선형 스캔** |
| 스킵 단위 | 행 단위 | 블록/파일 단위 (수천~수만 행) |
| 점 조회 | `WHERE k=X` 매우 빠름 | 블록 내부는 여전히 스캔 |
| "커버링" 의미 | 인덱스에 필요 컬럼 포함 → 테이블 미접근 | 해당 개념 없음 |

### 왜 다른가

**커버링 인덱스**: 별도 B-tree 자료구조. `WHERE x=5` → O(log n) 탐색 후 인덱스에 포함된 컬럼만으로 결과 완성. **인덱스만 읽고 테이블 안 읽음** 이 핵심.

**클러스터링**: 테이블 데이터 자체를 정렬 + 블록별 min/max 메타데이터. 쿼리 시 관련 없는 블록 통째로 스킵하고 나머지는 선형 스캔.

즉 **목적이 다름**:
- 커버링 인덱스 = "인덱스만 보고 답한다" (정확 조회 빠르게)
- 클러스터링 = "관련 없는 블록은 건너뛴다" (대규모 스캔 비용 감축)

### BigQuery 에서 "커버링" 개념이 불필요한 이유

BQ 는 **columnar store** — 각 컬럼이 별도 파일로 저장.

```sql
SELECT win, champion_id FROM match WHERE tier_bucket = 8000
```
이 쿼리는 `win`, `champion_id`, `tier_bucket` 3개 컬럼 파일만 읽음. 다른 컬럼은 원래부터 디스크 접근 안 함.

즉 columnar 스토리지 자체가 **"모든 컬럼 조합의 커버링"을 무료 제공**. BQ 설계 시에는 "어떤 컬럼 조합의 커버링 인덱스를 만들지" 가 아니라 "**어떤 컬럼으로 클러스터링해야 블록 스킵이 잘 될지**" 를 고민하게 됨.

## 요약

- **클러스터링 = 블록 단위의 근사 스킵 메커니즘** (인덱스 아님)
- 커버링 인덱스처럼 O(log n) 탐색은 못 하지만, 대규모 테이블에서 불필요한 블록을 버리는 데 매우 효과적
- **파티셔닝과 조합** 해서 써야 최고 효과 (파티션 프루닝 → 블록 스킵 → 선형 스캔)
- 클러스터 컬럼 순서/선택도 설계가 성능의 핵심
